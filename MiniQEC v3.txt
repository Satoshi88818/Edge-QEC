""" def __init__(self, distance: int = 3): if distance != 3: raise NotImplementedError("Only distance-3 surface code implemented. """ n = self.code.num_qubits m = self.code.num_stabilizers correction = np.zeros(n, dtype=int) # For repetition code: syndrome[i]=1 means boundary between qubits i, i+1 # Find defect positions and pair adjacent ones defects = [i for i in range(m) if syndrome[i] == 1] # Pair adjacent defects greedily i = 0 while i < len(defects) - 1: a, b = defects[i], defects[i + 1] # Flip all qubits between defect a+1 and defect b for q in range(a + 1, b + 1): if q < n: correction[q] ^= 1 i += 2 return correction def logical_error_rate(self, code: StabilizerCode, physical_error_rate: float, num_trials: int = 1000) -> float: """ Estimate logical error rate at a given physical error rate. Check-to-variable update (min-sum approximation) for i in range(m): neighbors = np.where(H[i])[0] if len(neighbors) == 0: continue for j in neighbors: others = [k for k in neighbors if k != j] if not others: c2v[i, j] = 0 continue sign = syndrome[i] if i < len(syndrome) else 0 product_sign = (-1) ** sign for k in others: product_sign *= np.sign(v2c[i, k]) if v2c[i, k] != 0 else 1 min_mag = min(abs(v2c[i, k]) for k in others) c2v[i, j] = product_sign * min_mag # Key Design Decisions

Why separate error_x and error_z in the environment? Real quantum errors are Pauli errors — X (bit-flip), Z (phase-flip), or Y (both). v2 only tracked bit-flip errors. v3 tracks both, enabling surface code correction.

Why not just always use Stim? Stim requires a C extension. The numpy fallback gives the same qualitative behaviour for small codes and keeps the library installable without a compiler.

Why is MWPM expected to beat RL on the repetition code? MWPM is provably optimal for the repetition code. RL's advantage emerges with device-specific noise models, non-Markovian errors, or when the syndrome extraction itself is noisy in structured ways that a learned decoder can exploit.

Why coremltools over torch.jit for iOS? coremltools produces a .mlpackage that Xcode integrates natively, includes metadata, and enables Apple Neural Engine offload via ComputeUnit.ALL. Direct TorchScript export would require the LibTorch runtime (~150MB) vs Core ML's on-device inference framework (~0 overhead, it's built into iOS).
